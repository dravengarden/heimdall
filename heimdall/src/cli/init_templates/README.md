# /etc/heimdall — config reference

Auto-generated by `heimdall init`. Regenerate after upgrading heimdall:

```bash
sudo /usr/local/bin/heimdall init --dir /etc/heimdall --format nickel --force
```

This README plus `lib.ncl` are the AI-readable contract for the
config schema. They version-lock with the daemon — if you upgrade
heimdall, regenerate both.

> **Audience**: this file is dense on purpose. It's the reference an
> AI agent reads when asked "edit /etc/heimdall/heimdall.ncl to do
> X". The user-facing how-to ("label your pod") lives elsewhere
> (typically a project-side doc that explains which connection names
> are wired up in this deployment).

## Directory layout

```
/etc/heimdall/
├── heimdall.ncl    ← main config (loaded by daemon)
├── lib.ncl         ← Nickel schema contracts (imported by heimdall.ncl)
├── README.md       ← this file
└── secrets/        ← 0700 root:root, NOT in git
    └── *.pw        ← 0400 root:root, no trailing newline
```

The daemon probes `/etc/heimdall/heimdall.{ncl,toml,json,yaml}` in
that order and uses the first one that exists, so a typical systemd
unit is just:

```
ExecStart=/usr/local/bin/heimdall serve
```

Override with `--config <PATH>` or `HEIMDALL_CONFIG=<PATH>` if you
keep the file elsewhere.

Format selection is by extension (`.yaml` / `.json` / `.toml` /
`.ncl`). Nickel is recommended because `lib.ncl` validates the whole
record at evaluation time — typos and wrong types are caught before
the daemon ever sees the file. Other formats are validated by
`serde(deny_unknown_fields)` at load time.

## Bootstrap workflow

```bash
# 1. Initialise (writes heimdall.ncl + lib.ncl + this README)
sudo /usr/local/bin/heimdall init --dir /etc/heimdall --format nickel

# 2. Edit /etc/heimdall/heimdall.ncl
sudoedit /etc/heimdall/heimdall.ncl

# 3. (Recommended) Validate before restart
nix-shell -p nickel --run "cd /etc/heimdall && nickel export -f json heimdall.ncl > /dev/null"

# 4. Add any required secrets
sudo install -d -m 0700 -o root -g root /etc/heimdall/secrets
printf '%s' 'PASSWORD' | sudo tee /etc/heimdall/secrets/<name>.pw > /dev/null
sudo chmod 0400 /etc/heimdall/secrets/<name>.pw

# 5. Apply
sudo systemctl restart heimdall
sudo journalctl -u heimdall -n 5 --no-pager | grep "config loaded"
# expected:
#   config loaded path=/etc/heimdall/heimdall.ncl connections=N pod_rules=M ...
```

`heimdall init --force` overwrites existing files. Without `--force`
it bails on any pre-existing target. It does **not** touch
`secrets/`.

## Top-level structure

```nickel
let h = import "lib.ncl" in
{
  apiVersion = "heimdall.io/v1alpha1",
  kind       = "HeimdallConfig",

  runtime     = { ... },              # eBPF + listener + retention
  connections = { name = { ... } },   # named SOCKS5 / direct upstreams
  podRouting  = {                     # pod selector → connection name
    routingKey = "heimdall.io/routing",
    observeKey = "heimdall.io/observe",
    rules      = [ ... ],
    "default"  = { use = "...", observe = ... },
  },
} | h.HeimdallConfig
```

`apiVersion` and `kind` are reserved sentinel strings. They must be
`"heimdall.io/v1alpha1"` and `"HeimdallConfig"` respectively in the
current schema.

## `runtime`

| Key | Type | Default | Meaning |
|---|---|---|---|
| `cgroup` | string | `/sys/fs/cgroup` | Cgroup root for `connect4` + `skb_egress` BPF attach. Use `/sys/fs/cgroup/kubepods` to scope to k8s. |
| `listen` | string | `0.0.0.0:12345` | Relay TCP listener (target of connect4 redirect). |
| `relayIp` | string | `127.0.0.1` | IPv4 the relay listens on as seen from pods. On k0s use `cilium_host` (`10.244.0.41`). |
| `bypassCidrs` | array of string | `[]` | Reserved; not yet wired into eBPF. Built-in bypass list covers loopback, link-local, LAN, k0s pod/svc CIDRs. |
| `dnsListen` | string | `0.0.0.0:5358` | Fake-IP DNS UDP listener. |
| `fakeIpCidr` | string | `198.19.0.0/16` | Fake-IP pool (RFC 2544 benchmark range, guaranteed not in real routing). |
| `apiListen` | string | `127.0.0.1:9999` | HTTP API + Web UI. Set `0.0.0.0:9999` for LAN access. |
| `stateDir` | string | `/var/lib/heimdall` | sqlite + future blob storage. systemd creates 0750 root:root. |
| `flowRetentionSecs` | integer | `259200` (3d) | Cleanup task drops `flows` + `messages` older than this. |
| `tap.enabled` | bool | `false` | Master switch for Phase B uprobe tap. When false: no /proc scan, no uprobes, no perf consumer. |
| `tap.persist` | bool | `false` | Within tap, write captured plaintext to the `messages` table. When false, events only show up in journal. |

`tap.enabled=false` overrides `tap.persist`. Tap requires
`CAP_SYS_PTRACE` in the systemd unit (already set in NixOS module).

## `connections`

A registry of named upstreams. Keys are the connection names
referenced from `podRouting.rules[].use` and from per-pod
annotation/label overrides.

```nickel
connections = {
  default = {
    description = "Public internet egress (e.g. local SOCKS5 proxy)",
    type        = "socks5",
    addr        = "127.0.0.1:1080",
  },

  # Example: a second upstream for a different egress path. Pick any
  # name your charts will reference — keys with hyphens or dots must
  # be quoted in Nickel record syntax (`"corp-vpn" = ...`); use a
  # plain identifier (`corpVpn` / `internal`) to avoid the quoting.
  internal = {
    description = "Auth-protected SOCKS5 (e.g. corporate VPN gateway)",
    type        = "socks5",
    addr        = "10.0.0.1:1080",
    auth = {
      username     = "alice",
      passwordFile = "/etc/heimdall/secrets/internal.pw",
    },
  },

  direct = { type = "direct" },
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `type` | enum `"socks5"` \| `"direct"` | yes | |
| `addr` | string | for `socks5` | host:port. |
| `auth.username` | string | optional | RFC 1929 user/pass. |
| `auth.passwordFile` | string | optional | Path read at daemon startup. No trailing newline. |
| `description` | string | optional | Surfaced in API + UI. |
| `owner` | string | optional | Free-form contact. |
| `mitm` | bool | optional | Reserved for M5 (TLS terminator). Parsed but ignored. |

### Reserved name `system`

`system` is **not** a connection — it's a reserved tag used in
`podRouting` to mean "no relay redirect; eBPF lets the connection
through untouched." The validator rejects a connection literally
named `system`.

Pods routed to `system` skip the relay entirely. They still go
through the v2raya host TPROXY (host-side) — heimdall just doesn't
intercept.

### Sentinel `default`

There must be a connection named `default`. It's the safety net
when `podRouting.default.use` is left unspecified or when an
override path resolves to a non-existent name and falls through.

## `podRouting`

```nickel
podRouting = {
  routingKey = "heimdall.io/routing",   # annotation/label key for `use` override
  observeKey = "heimdall.io/observe",   # annotation/label key for `observe` override
  rules      = [ ... ],                 # ordered list, first match wins
  "default"  = { use = "...", observe = bool },
}
```

`"default"` is quoted because `default` is a reserved word in
Nickel. The runtime field name is `default`.

### Resolution order (each axis independently)

```
USE / CONNECTION:
  1. pod.annotations[routingKey]
  2. pod.labels     [routingKey]
  3. first matching rule in podRouting.rules
  4. podRouting.default.use

OBSERVE:
  1. pod.annotations[observeKey]   ← parsed: true|1|yes|on / false|0|no|off
  2. pod.labels     [observeKey]   ← same parser
  3. first matching rule in podRouting.rules (only if rule defines observe)
  4. podRouting.default.observe
```

The two axes are **fully independent** — a pod can pick
`use: system` (skip relay) and still have `observe: true` (capture
plaintext via uprobes). A pod can route via the relay and have
observe suppressed.

### Rules

```nickel
rules = [
  {
    name    = "string",          # optional; appears in logs
    "match" = MatchCond,         # optional; absent = catchall
    use     = "default",         # connection name, or reserved "system"
    observe = true,              # optional; omit to inherit podRouting.default.observe
  },
  ...
]
```

`"match"` is quoted because `match` is a reserved word in Nickel.

`use` MUST be one of:
- A name present in `connections` (case-sensitive, exact match)
- The reserved string `"system"`

Validation rejects unknown names at startup.

`observe` is optional. If omitted, the rule contributes only to
`use`; the observe axis falls through to default.

Rules are evaluated **in order, first match wins**. Put more
specific rules above more general ones.

## `MatchCond` — the selector grammar

K8s LabelSelector compatible plus boolean composition.

```nickel
{
  # Pod selectors
  namespaces       = [ "ns-a", "ns-b" ],         # OR over namespace
  matchLabels      = { key = "value", ... },     # AND of equality
  matchExpressions = [ MatchExpression, ... ],   # AND of expressions

  # Boolean composition — every leaf is itself a MatchCond
  all = [ MatchCond, ... ],   # AND
  any = [ MatchCond, ... ],   # OR
  "not" = MatchCond,          # NOT  (`not` reserved in Nickel; quote it)
}
```

All present **fields** AND together. So:

```nickel
{
  namespaces  = [ "opik" ],
  matchLabels = { "app.kubernetes.io/name" = "opik" },
}
```

…matches "namespace is opik AND label app.kubernetes.io/name=opik".

An empty MatchCond (or `all = []`) matches everything. Omit `match`
on a rule to make it a catchall.

### `MatchExpression` operators

```nickel
{
  key      = "string",
  operator = "In" | "NotIn" | "Exists" | "DoesNotExist",
  values   = [ "v1", "v2", ... ],   # required for In / NotIn; omit for Exists / DoesNotExist
}
```

| Operator | `values` | Semantics |
|---|---|---|
| `In` | required, ≥ 1 | Label exists AND its value is in the list. |
| `NotIn` | required, ≥ 1 | Label is missing OR its value is NOT in the list. |
| `Exists` | absent / empty | Label key is present (any value). |
| `DoesNotExist` | absent / empty | Label key is not set on the pod. |

### Boolean composition example

"namespace is `payments` AND (label X=foo OR label Y is one of [a, b])":

```nickel
{
  namespaces = [ "payments" ],
  any = [
    { matchLabels = { X = "foo" } },
    { matchExpressions = [
        { key = "Y", operator = "In", values = [ "a", "b" ] }
      ],
    },
  ],
}
```

"NOT in kube-system":

```nickel
{
  "not" = { namespaces = [ "kube-system" ] },
}
```

## `cli` — defaults for `heimdall <subcommand>`

Optional top-level field that lets every CLI default live in the
same config file as routing. No `~/.config/heimdall/cli.toml`,
nothing else to discover. Each subcommand hangs its config under
`cli.<subcmd>`. Today only `cli.run` is consumed (by the planned
proxychains-style `heimdall run`).

```nickel
cli = {
  run = {
    # Baseline applied when --profile is not given. Every field optional.
    "default" = {
      connection  = "default",       # connection name (or reserved "system")
      observe     = true,             # capture plaintext via the tap
      dns         = "fake",           # "fake" | "system" — DNS strategy
      timeout     = 0,                 # seconds; 0 = no timeout
      extraBypass = [],                # extra CIDRs to bypass for this command
      tag         = "",                # free-form label, surfaces in flow log
    },

    # Named profiles selectable via --profile NAME. Inherits from
    # default; override only the fields you care about.
    profiles = {
      internal = { connection = "internal" },
      peek     = { tag = "interactive-debug" },
      silent   = { observe = false },
    },
  },
}
```

### Resolution order for `heimdall run`

```
flag (e.g. --connection X)
  > [profile.NAME] in cli.run.profiles
  > cli.run.default
  > compiled-in fallback (connection="default", observe=true, dns="fake")
```

Adding more subcommand defaults later is non-breaking — just add a
new `cli.<other-subcmd>` entry. All `cli.*` fields are optional, so
existing configs continue to validate.

## Per-pod overrides (no config edit needed)

Set on the pod's metadata:

```yaml
metadata:
  annotations:
    heimdall.io/routing: internal  # pick a connection by name
    heimdall.io/observe: "true"    # force observe regardless of rule
  labels:
    heimdall.io/routing: internal  # labels work too; annotations win if both set
```

Annotation > label > rule > default. Annotations are usually a
better choice because they're harder to set accidentally via Helm
templating.

`heimdall.io/observe` parses (case-insensitive):
`true|false`, `1|0`, `yes|no`, `on|off`. Anything else falls
through to the next resolution layer.

## Reserved fields and edge cases

| Item | Behaviour |
|---|---|
| `apiVersion` != `"heimdall.io/v1alpha1"` | Validation error at load. |
| `kind` != `"HeimdallConfig"` | Validation error at load. |
| `connections.system` | Rejected (reserved tag). |
| `podRouting.rules[].use = "non-existent"` | Rejected at startup. |
| `connections.<name>` missing for `auth.passwordFile` | Daemon startup fails reading the file. |
| Empty `podRouting.rules` | OK; everything falls to `podRouting.default`. |
| `podRouting.default.use = "system"` | OK (everything skips the relay by default). |
| Unknown field anywhere (Nickel) | Caught by `lib.ncl` contract at evaluation. |
| Unknown field anywhere (YAML/JSON/TOML) | Caught by `serde(deny_unknown_fields)` at load. |

## Validation — what the daemon checks at startup

Order of operations:

1. Load + parse the file according to its extension.
2. Run schema-level validation (`serde` for non-Nickel; `lib.ncl`
   for Nickel runs at file-evaluation time before this point).
3. Validate cross-references:
   - Every `podRouting.rules[].use` exists in `connections` or
     equals `"system"`.
   - `podRouting.default.use` similarly.
4. Read every `connections.<name>.auth.passwordFile`.
5. Reconcile pod policy from current pod cache (informer hasn't
   synced yet on first startup; it logs a warning and uses the
   default decision until sync completes).

Failures at any step abort startup — heimdall does NOT serve a
partial config.

## Smoke test

After any change, verify the routing decision via the flows API:

```bash
# 1. Generate traffic from a pod that should match the rule under test
sudo KUBECONFIG=/var/lib/k0s/pki/admin.conf kubectl exec -n <ns> <pod> -- \
  curl -ksSm 5 -o /dev/null https://www.cloudflare.com/

# 2. Inspect the resulting flow
curl -sS "http://127.0.0.1:9999/api/flows?limit=20" | python3 -c '
import json, sys
for f in json.load(sys.stdin):
    if f["connection_name"] == "bypass": continue
    dst = f["dst_host"] or f["dst_ip"]
    print(f"ns={f[\"namespace\"]:20s} pod={f[\"pod_name\"]:50s} → {f[\"connection_name\"]:10s} {dst}:{f[\"dst_port\"]} atyp={f[\"atyp\"]}")
'
```

Expected:
- `connection_name` matches the `use:` your rule resolved to.
- `dst_host` non-empty (`atyp=domain`) when fake-IP DNS round-tripped.
- `dst_host` null with IP literal (`atyp=ip`) when the pod connected
  by raw IP — the relay handled it via SOCKS5 ATYP=0x01.

For pods routed to `use: system`, **no flow record appears** (eBPF
lets the connection skip the relay entirely).

For traffic where `dst_ip` is in the cluster CIDR or the bypass
list, you'll see `connection_name: "bypass"` — that's the synthetic
log the eBPF bypass path emits, not a real relay flow.

### Smoke test without a live pod

When the cluster is unhealthy, put a process directly in the
kubepods cgroup — eBPF treats it identically to a pod:

```bash
sudo mkdir -p /sys/fs/cgroup/kubepods/heimdall-test
sudo bash -c 'echo $$ > /sys/fs/cgroup/kubepods/heimdall-test/cgroup.procs \
  && curl -sS -m 5 https://www.cloudflare.com/ -o /dev/null -w "%{http_code}\n"'
sudo rmdir /sys/fs/cgroup/kubepods/heimdall-test
```

This exercises `podRouting.default` (no pod = no labels = no
namespace match). Result `200` confirms relay → upstream → internet
chain is healthy.

## Common errors

| Symptom | Cause | Fix |
|---|---|---|
| `unknown field 'namespace', expected 'namespaces' …` | Old singular field | Rename `namespace:` → `namespaces:` (now an array). |
| `unknown use 'foo' in pod rule` at startup | Misspelled or removed connection | Fix `use:` to a real connection name or `"system"`. |
| `connections registry: 'system' is reserved` | Declared `connections.system` | Drop that entry; use `use: system` in a rule instead. |
| `cannot read passwordFile: ...` | Missing or wrong-permissions secret | `chmod 0400` and `chown root:root /etc/heimdall/secrets/<name>.pw`. |
| All flows show `connection_name: "bypass"` | All traffic is intra-cluster (10.96/12, 10.244/16) — expected | Generate external traffic to test relay path. |
| `nickel: command not found` outside systemd | `nickel` not on user PATH | `nix-shell -p nickel --run ...` or `nix profile install nixpkgs#nickel`. |
| CoreDNS forward target reverts | k0s rewrites coredns ConfigMap on restart | The `k0s-coredns-patch.service` re-applies; don't edit by hand. |

## Schema source

| Layer | File |
|---|---|
| Rust types | `heimdall-config/src/lib.rs` |
| Nickel contracts | `lib.ncl` (this directory) |
| Init templates | `heimdall/src/cli/init_templates/` in repo |

When the daemon version changes, regenerate `lib.ncl` and this
README via `heimdall init --force`. The Rust types are the source
of truth; `lib.ncl` and the embedded templates are kept in sync by
the heimdall release process.
