# Editing /etc/heimdall/heimdall.{ncl,yaml,json,toml}

The single config file declares everything heimdall does:

```
runtime     — eBPF + listener + retention knobs
connections — named SOCKS5 / direct upstreams
podRouting  — pod-selector → connection name (annotation/label/rule/default)
cli         — defaults for `heimdall <subcommand>` invocations
```

## Authoritative schema reference

**Always read `/etc/heimdall/README.md` before editing.** It's
auto-generated by `heimdall init` and version-locked to the daemon
binary. It documents every field, every operator, every default,
plus validation rules and common errors.

Don't guess from this skill — it's an entry-point pointer, not the
spec.

## Recommended format: Nickel

`heimdall.ncl` is recommended because it imports `lib.ncl` and
validates the whole record at evaluation time. Typos (e.g.
`namespace` instead of `namespaces`) and wrong types fail at
`nickel export`, before the daemon ever loads the file.

YAML / JSON / TOML are accepted (extension-detected), validated by
`serde(deny_unknown_fields)` at daemon load time.

## Workflow

```bash
# 1. Read the schema reference
less /etc/heimdall/README.md

# 2. Edit
sudoedit /etc/heimdall/heimdall.ncl

# 3. (Nickel only) validate locally
nix-shell -p nickel --run "cd /etc/heimdall && nickel export -f json heimdall.ncl > /dev/null"

# 4. Apply
sudo systemctl restart heimdall
heimdall status                          # verify connections=N pod_rules=M
heimdall flows search --since 1m --json  # smoke-test a routing decision
```

## Common edits

### Add a new SOCKS5 upstream

In `heimdall.ncl`:

```nickel
connections = {
  default  = { ... },
  myproxy  = {
    description = "...",
    type        = "socks5",
    addr        = "10.0.0.5:1080",
    auth = {                          # optional
      username     = "alice",
      passwordFile = "/etc/heimdall/secrets/myproxy.pw",
    },
  },
  ...
}
```

Then add the password file:

```bash
printf '%s' 'PASSWORD' | sudo tee /etc/heimdall/secrets/myproxy.pw > /dev/null
sudo chmod 0400 /etc/heimdall/secrets/myproxy.pw
sudo chown root:root /etc/heimdall/secrets/myproxy.pw
```

### Route a namespace / label-selector to that upstream

Append to `podRouting.rules` (first match wins, top-down):

```nickel
{
  name    = "my-rule",
  "match" = {
    namespaces = [ "tenant-a" ],
    matchLabels = { "app.kubernetes.io/part-of" = "tenant-a" },
  },
  use     = "myproxy",
  observe = true,                     # capture plaintext via tap
}
```

For boolean composition (`all` / `any` / `not`) and the four
`matchExpressions` operators (`In` / `NotIn` / `Exists` /
`DoesNotExist`), see `/etc/heimdall/README.md`.

### Per-pod override (no config edit)

Set on the pod's metadata — annotation wins over label, both win
over rules:

```yaml
metadata:
  annotations:
    heimdall.io/routing: myproxy
    heimdall.io/observe: "false"
```

### Change `heimdall run` defaults

```nickel
cli = {
  run = {
    "default" = {                     # baseline (no --profile flag)
      connection = "default",
      observe    = true,
      dns        = "fake",
    },
    profiles = {
      corp = { connection = "corp" },
      silent  = { observe = false },
    },
  },
}
```

Resolution: `flag > [profile.NAME] > cli.run.default > compiled-in fallback`.

## Reserved values

- Connection name `system` is **reserved**. Don't declare
  `connections.system`. Use `use: "system"` in a rule to mean
  "skip the eBPF redirect, no relay involvement."
- Connection name `default` is **required**. Validation fails
  without it.

## Validation gates

The daemon refuses to start when:
- `apiVersion` ≠ `"heimdall.io/v1alpha1"`
- `kind` ≠ `"HeimdallConfig"`
- `connections` doesn't include `default`
- Any rule's `use` references an unknown connection (and isn't `"system"`)
- A `passwordFile` is unreadable

All of these surface in `journalctl -u heimdall` immediately on
restart.
