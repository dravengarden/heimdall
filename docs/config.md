# Routing pods through heimdall

How to make a Kubernetes workload send its egress through a specific
heimdall connection (Corp VPN, public internet, eBPF bypass, …).
**This file is K8s-side only** — no heimdall internals. For the full
schema and admin-side editing, see `/etc/heimdall/README.md` on the
the host (auto-generated by `heimdall init`).

> Single source of truth: this file lives at
> `~/heimdall/docs/config.md` and is symlinked into
> `/etc/<host-config>/docs/services/heimdall-config.md` and
> `~/corp/docs/heimdall.md`. Edit here.

## What heimdall gives a pod

Per-pod choice of:

1. **Which upstream egresses your traffic** — selected by a
   *connection name* (string, e.g. `default`, `corp`, `system`).
2. **Whether plaintext is captured** — observability flag.

Both are decided independently by the heimdall control plane on the
the host. As a chart author you don't see proxy URLs, credentials,
or VPN details — only connection names.

## Connection names available on the host

Stable contract owned by the the admin (see `/etc/heimdall/heimdall.ncl`).

| Name | Egresses to | When to use |
|---|---|---|
| `default` | Public internet via local v2raya | github.com, npm, pypi, docker hub, public cloud APIs — anything that doesn't need the corporate VPN. **This is the cluster-wide default**, you only need to set it explicitly for clarity. |
| `corp` | Mac SOCKS5 (`<UPSTREAM_IP>:1080`) → AnyConnect VPN | Anything ending in `.corp.ai`, Corp-internal git/registries, internal cluster APIs, IP literals inside corp VPN address spaces. |
| `direct` | Direct connect, no upstream proxy | Rarely needed. Useful if a pod needs the relay's logging but not the SOCKS5 layer. |
| `system` | **Skip heimdall entirely** (no eBPF redirect) | Cluster-internal infra that must not pass through any user-space proxy. Already covered by the `cluster-infra` admin rule for `kube-system` + `local-path-storage`. Don't set on application pods. |

## How to select a connection

Three mechanisms, **highest priority wins**:

### 1. Pod annotation (preferred for chart-author overrides)

```yaml
metadata:
  annotations:
    heimdall.io/routing: corp
    heimdall.io/observe: "true"   # optional; force observability
```

### 2. Pod label (declarative, queryable by other tooling)

```yaml
metadata:
  labels:
    heimdall.io/routing: corp
```

Annotations win over labels if both are set.

### 3. Admin-side rule (no pod change required)

If you can't modify the chart, the the admin can add a `podRouting`
rule that matches by namespace / labels. Open a request — see
`/etc/heimdall/heimdall.ncl` for the rule list and append a new one.

### Resolution order

```
annotation > label > admin rule > admin default (default = "default")
```

## Helm-chart patterns

### Override only the egress connection

```yaml
# values.yaml
podAnnotations:
  heimdall.io/routing: corp
```

### Apply to selected pods within a chart

```yaml
# values-corp.yaml
podLabels:
  heimdall.io/routing: corp

# OR per-template
spec:
  template:
    metadata:
      annotations:
        heimdall.io/routing: corp
```

### Suppress observability for a sensitive pod

```yaml
podAnnotations:
  heimdall.io/observe: "false"
```

`heimdall.io/observe` accepts (case-insensitive):
`true|1|yes|on` and `false|0|no|off`. Anything else falls through to
the admin default.

## Verifying it worked

After deploying the labelled pod, check the routing decision via the
heimdall API on the the host:

```bash
# Generate egress traffic from the pod under test
sudo KUBECONFIG=/var/lib/k0s/pki/admin.conf kubectl exec -n <ns> <pod> -- \
  curl -ksSm 5 -o /dev/null https://www.cloudflare.com/

# Inspect what connection was picked
curl -sS "http://<HOST_IP>:9999/api/flows?limit=20" | python3 -c '
import json, sys
for f in json.load(sys.stdin):
    if f["connection_name"] == "bypass": continue
    dst = f["dst_host"] or f["dst_ip"]
    print(f"{f[\"namespace\"]:20} {f[\"pod_name\"]:50} → {f[\"connection_name\"]:10} {dst}")
'
```

You should see a row with the connection name you asked for.

| Result | Meaning |
|---|---|
| `connection_name = "corp"` | Routing took effect. |
| `connection_name = "default"` | Override didn't apply — annotation/label key may be wrong, or admin rule overrode. |
| No row | Either the pod made no egress requests yet, **or** routing resolved to `system` (eBPF skipped the relay), **or** the destination is in the bypass list (cluster CIDR / loopback / LAN / fake-IP pool). |

For a deeper inspection (per-pod cgroup_id, fake-IP mapping,
plaintext messages from the tap), point a browser at the heimdall UI:
`http://<HOST_IP>:9999/`.

## Pitfalls

- **Don't set `heimdall.io/routing: system` on application pods.**
  That bypasses the relay entirely — your pod gets *no* heimdall
  features (no SOCKS5 routing, no observability, no fake-IP DNS).
  Reserved for cluster infra.
- **Don't expect the hostname inside the pod to resolve via Mac
  DNS.** The pod's libc resolver hits CoreDNS → fake-IP DNS, which
  returns synthetic IPs from `198.19.0.0/16`. Heimdall reverses the
  lookup at relay time and sends the hostname over SOCKS5
  (ATYP=0x03). The Mac then resolves via its own DNS chain
  (including AnyConnect-pushed scoped resolvers).
- **Cluster-internal traffic doesn't go through heimdall.** Pod ↔
  pod (`10.244.0.0/16`) and pod ↔ Service (`10.96.0.0/12`) are
  bypassed by eBPF. You don't need to do anything for them.
- **Don't try to add a new connection from the chart side.** New
  upstream registrations are admin-only — they require credentials
  on the the host. Open an issue / PR against `services/heimdall/`
  in the NixOS repo and `/etc/heimdall/heimdall.ncl` on the host.
